# M2 Group 1: Notes Layer (Revised for M1 Reality)

## Context

The M2 plan `2026-04-17-petagent-m2-skill-evolution.md` Group 1 (Tasks 1-5) was authored before M1 shipped. Two things drifted:

1. **Schema gap.** M1 migration `0058_careless_morlun.sql` created `agent_notes` with columns `id, company_id, agent_id, issue_id, session_id, note_type, body, tags, metadata, git_commit_sha, created_at`. The M2 Task 2 code references `embedding` and `scope` columns that don't exist.
2. **API gap.** `GitStore.writeFile(relPath, content, message)` on the `@petagent/safety-net` package takes a 3-arg signature, while the M2 plan calls `store.write(relPath, content, { author, reason, scope })`.

This spec revises Group 1 to match M1 reality. The original M2 plan stays as the high-level reference; this document is the authoritative specification for the Notes layer implementation.

`pgvector` extension is already enabled by migration 0058 (`CREATE EXTENSION IF NOT EXISTS vector`), so no new extension is needed.

---

## 1. Architecture

The Notes layer adds three concerns on top of M1 primitives:

- **Persistence**: agent notes are written to a `GitStore` (file + commit) AND to the `agent_notes` Postgres table. The Git copy is the source of truth for human inspection / git history; the DB copy carries embedding + structured metadata for semantic search.
- **Embedding**: a pluggable `EmbeddingService` produces 1536-dim vectors. M2 MVP ships a deterministic hash stub; Task 30a (separate, later in M2) wires the real API.
- **Search**: `NotesManager.search` uses `pgvector` cosine distance (`embedding <=> $1::vector`).

Files added under `packages/skills/src/` so consumer surface stays a single import. (The original M2 plan also kept this colocation.)

```
packages/skills/src/
  embedding.ts          # EmbeddingService (stub-backed for MVP)
  notes_manager.ts      # NotesManager.create / get / list / search
packages/skills/src/__tests__/
  embedding.test.ts
  notes_manager.test.ts
packages/db/src/schema/agent_notes.ts   # add embedding + scope columns
packages/db/src/migrations/<next>.sql   # generated by drizzle-kit
packages/cli/src/commands/notes.ts      # CLI list/search/view
server/src/routes/agent-notes.ts        # route powering the CLI
```

Tests under `packages/skills/src/__tests__/` follow the same convention as Tasks 1-8 of #1d, kebab-case `.test.ts`.

## 2. Schema Additions

`agent_notes` gains two columns:

- `embedding vector(1536)` — nullable. Embeddings are produced on insert; nullable so historical rows from M1 can stay valid until back-filled.
- `scope text NOT NULL DEFAULT 'project'` — values `'user' | 'project' | 'local'`. Constrained at the application layer (Zod), not via a CHECK constraint, to keep migration light.

A second pgvector index for `embedding` is added in the same migration:

```sql
CREATE INDEX agent_notes_embedding_idx ON agent_notes
  USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);
```

`lists = 100` is a defensible default for `<= 10K` notes per company; re-tuning at scale is a deferred concern (the original M2 plan flagged this).

## 3. `EmbeddingService`

```ts
export interface EmbeddingServiceOptions {
  apiKey?: string;
  model?: string;
  /** When true (default in test env, false in prod), use the deterministic stub. */
  useStub?: boolean;
}

export class EmbeddingService {
  constructor(opts?: EmbeddingServiceOptions);
  embed(text: string): Promise<number[]>;
  embedBatch(texts: string[]): Promise<number[][]>;
}
```

- **Stub semantics.** Deterministic 1536-dim vector derived from the SHA-256 hash of the input text, normalized to unit length. Same input → same vector → semantic-search tests get stable ordering. Different inputs map to different vectors via different hash bytes; cosine similarity is meaningful enough for unit tests, not for production retrieval.
- **`useStub` resolution**: explicit `opts.useStub` wins. If unset, fall back to `opts.apiKey === undefined ? true : false` — i.e., no key means stub mode. Tests construct `new EmbeddingService()` and get the stub. Production code in Task 30a passes `{ apiKey, useStub: false }`. Constructing `new EmbeddingService({ useStub: false })` without `apiKey` throws.
- **Real API call**: a private `callEmbedAPI` method is declared but throws `Not implemented yet (Task 30a)` so the contract is visible without the wiring.

## 4. `NotesManager`

```ts
export type NoteScope = "user" | "project" | "local";

export interface CreateNoteArgs {
  agentId: string;
  content: string;
  scope: NoteScope;
  tags?: string[];
  noteType?: string;        // free-form e.g. "reflection", "lesson", "insight"; default "note"
  sourceIssueId?: string | null;
  sessionId?: string | null;
  metadata?: Record<string, unknown>;
}

export interface NoteRecord {
  id: string;
  agentId: string;
  companyId: string;
  scope: NoteScope;
  content: string;
  tags: string[];
  noteType: string;
  embedding: number[] | null;
  gitCommitSha: string | null;
  createdAt: Date;
  metadata: Record<string, unknown>;
}

export class NotesManager {
  constructor(deps: {
    db: Db;
    embedder: EmbeddingService;
    store: GitStore;
    companyId: string;
  });

  create(args: CreateNoteArgs): Promise<NoteRecord>;
  get(id: string): Promise<NoteRecord | null>;
  list(args: { agentId: string; limit?: number }): Promise<NoteRecord[]>;
  search(args: { agentId: string; query: string; topK?: number; scope?: NoteScope }): Promise<NoteRecord[]>;
}
```

Behavior:

- **`create`**: embed the content → write a markdown file via `GitStore.writeFile(relPath, body, message)` where:
  - `relPath = "agents/{agentId}/notes/{YYYY-MM-DD}-{slug}.md"`
  - `body` is markdown with YAML frontmatter (`scope`, `tags`, `noteType`, `sourceIssueId`)
  - `message = "note({scope}/{noteType}): {content[:60]}..."`
  - then INSERT the row with `embedding`, `scope`, `git_commit_sha = sha`, etc.
- **`get`**: SELECT by id, scoped by `companyId`. Returns null if missing.
- **`list`**: SELECT by `agentId + companyId`, ORDER BY `created_at DESC` LIMIT (default 50, max 200).
- **`search`**: embed the query, then SELECT with `embedding <=> $vec` ordered ASC, LIMIT topK (default 10, max 50). When `scope` is provided, AND-filter on it. Notes with NULL embedding are excluded (`WHERE embedding IS NOT NULL`).

Atomicity: `GitStore.writeFile` happens before the DB insert. If the DB insert fails, the file remains in git history (orphan but harmless). If the file write fails, no DB row is created. We do NOT add a compensating delete to git history — git is append-only and the orphan file is acceptable. Documented in code.

## 5. CLI: `petagent notes`

Three subcommands. All read-only from the CLI perspective (creation is reserved for the agent runtime).

```
petagent notes list --agent <id> [--limit N] [--scope user|project|local]
petagent notes search --agent <id> --query "..." [--topK N] [--scope ...]
petagent notes view <id>
```

The CLI calls a new server route:

```
GET  /api/companies/:companyId/agents/:agentId/notes?limit=N&scope=...
GET  /api/companies/:companyId/agents/:agentId/notes/search?q=...&topK=N&scope=...
GET  /api/notes/:id
```

Auth: agents can only access their own notes by default; board users can access any agent's notes within their company. Same shape as existing `agent-instructions` routes.

## 6. Testing

### `embedding.test.ts` (~5 tests)

- Stub returns a 1536-dim vector for any input.
- Same input → identical vector across calls.
- Different inputs → different vectors.
- `embedBatch` returns an array of vectors with matching length.
- `useStub: false` + no `apiKey` → constructor throws (caller must either stub or provide a key).

### `notes_manager.test.ts` (~6 tests)

- `create` writes file via `GitStore.writeFile` then DB row, returns the round-trip record with `embedding.length === 1536` and `gitCommitSha` populated.
- `create` rejects when scope is invalid (Zod-level).
- `get` returns null for unknown id.
- `list` returns most-recent first, respects limit.
- `search` returns top-K ordered by cosine distance ASC. Test corpus uses 5 hand-crafted notes; the search query embeds to the same vector as one of them (deterministic stub) and that note ranks first.
- `search` honors `scope` filter (note with `scope="user"` doesn't appear when `scope="project"` is passed).

### CLI tests (~3 tests)

Pure helper tests for `buildListUrl` / `buildSearchUrl` / `parseNoteListResponse` (similar to `buildHireFormDefaults` pattern from M1 Group 12).

### Integration test (~1 test)

End-to-end: in-memory db (embedded postgres), real `GitStore` against tmp dir, agent creates 5 notes, search returns expected order. Mirrors the structure of #1d's `psychologist-server-stack.test.ts` but with real persistence.

Total: ~15 new tests across Group 1.
